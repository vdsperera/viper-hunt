A new target spawns instantly on a random unoccupied grid cell the moment an active target is consumed."

### REQ-005
- **Type:** Gap
- **Location:** Section 5: "✦ Hunter resets to default starting position at the beginning of each new level"
- **Problem:** If the Hunter's body length is long when entering a new level, resetting the head position without defining the body tail configuration could cause immediate self-collision, instant death, or rendering off-screen.
- **Question:** How is the Hunter's extended body tail arranged or oriented on the grid when resetting to the starting position at the beginning of a new level?
- **Suggested fix:** Change to: "At the start of a new level, the Hunter's head is placed at the center grid coordinate (X_mid, Y_mid), and the remaining trailing body segments are arranged horizontally extending to the left. If the length exceeds available grid space, the tail wraps linearly along the grid perimeter without colliding."

### REQ-006
- **Type:** Conflict
- **Location:** Section 8: "✦ Time is measured per level... level_score = level_target_value_sum × value_weight + (1 / level_time_seconds) × time_weight"
- **Problem:** If a player completes a level in 0 seconds due to a frame glitch or edge execution, a division-by-zero error occurs (1/0). Furthermore, using linear reciprocal scaling means a difference between 1 and 2 seconds alters the score drastically, while the difference between 60 and 61 seconds is practically non-existent.
- **Question:** Can we implement a protected baseline duration or a standard linear time bonus subtraction model to prevent division-by-zero errors and normalize scoring?
- **Suggested fix:** Change to: "Level time tracking starts at 0.0 seconds and increments continuously. To prevent division-by-zero errors and balance scoring distributions, the time score component uses a linear countdown bonus: time_score = max(0, (Max_Level_Time − level_time_seconds)) × time_weight, where Max_Level_Time is a configurable integer (default: 300 seconds)."

## Refined Requirements

### 1. Platform

- **1.1 [REFINED]** MVP target platform is explicitly a web browser application built using standard HTML5 Canvas and JavaScript (ES6+), optimized for both desktop and mobile web browsers.
- **1.2 [REFINED]** The application layout uses a fixed canvas rendering resolution of 1280×720 pixels (16:9 aspect ratio) with dynamic responsive CSS viewport scaling.
- **1.3 [REFINED]** Offline play is supported via client-side caching of local code assets and an integrated local fallback database.

### 2. Actors & Roles

- **2.1 [REFINED]** **Player:** An end-user who interacts with the game via keyboard inputs or mobile touch D-Pad to control the direction of the Hunter through a 2D grid matrix.
- **2.2 [REFINED]** **System Admin:** A privileged user who modifies, appends, or deletes rows within a designated remote configuration spreadsheet (Google Sheets) to adjust game parameters and target definitions.

### 3. Criminal Registry & Play Data

- **3.1 [REFINED]** The canonical target database is maintained via a Google Sheet during the MVP phase. At application launch, the client fetches runtime target definitions from this sheet. If the connection fails or if the system is offline, the app falls back to a locally bundled `fallback_registry.json` asset.
- **3.2 [REFINED]** For the private internal MVP lifecycle, the registry is pre-seeded with synthetic data or public reference items. Fictionalized illustrated graphics are used for public builds.
- **3.3 [REFINED]** Each record must strictly contain: `Name`, `Avatar_Asset_Path`, `Interpol_Red_Notice`, `FBI_Most_Wanted`, `Conviction_Status`.
- **3.4 [REFINED]** A specific criminal record can spawn at most once per single gameplay session. Once captured, that specific record ID is removed from the active session pool.
- **3.5 [REFINED]** Modifications made to the Google Sheet by the Admin apply dynamically at the next application initialization or session reset.

### 4. Core Game Mechanic

- **4.1 [REFINED]** The game area operates on a discrete grid pattern of X × Y cells. The Hunter occupies a sequential chain of grid cells and automatically advances forward by 1 grid unit per game frame tick in the last registered cardinal direction (Up, Down, Left, Right).
- **4.2 [REFINED]** A gameplay session terminates instantly, triggering the Game Over sequence, if the head coordinate of the Hunter matches any coordinate of its own trailing body segments (self-collision), matches any coordinate outside the defined grid bounds (boundary collision), or intersects a roaming Big Boss hazard.
- **4.3 [REFINED]** Overlapping the Hunter's head coordinate with a target's coordinate triggers a capture event: the target is deleted, score & particle FX emit, and the Hunter's body segments grow based on the target value tier.
- **4.4 [REFINED]** Movement direction reversals of 180 degrees within a single tick frame are strictly rejected.

### 5. Level Structure

- **5.1 [REFINED]** Game progression is broken down into consecutive sequential levels. The moment the number of captured targets equals the total targets configured for the active level (N), the system increments the level counter and instantly renders the new level's targets without pausing.
- **5.2 [REFINED]** The Hunter's size (total trailing segments) does not reset when transitioning to a new level; the full structural length is maintained throughout the session.
- **5.3 [REFINED]** At the start frame of a new level, the Hunter's head coordinate is moved to the exact center grid coordinate (X_mid, Y_mid), and trailing body segments extend horizontally to the left.
- **5.4 [REFINED]** Levels are configured externally via `total_level_targets` and `max_simultaneous_targets`.

### 6. Target Value & Display

- **6.1 [REFINED]** Base value calculation:
  ```
  target_value = (interpol_red_notice × 40) + (fbi_most_wanted × 35) + (conviction_status × 25)
  ```
- **6.2 [REFINED]** In Mode 1 (Bounty Hunter), targets render avatar graphics with numerical value indicators. In Mode 2 (Treasure Vault), targets render glowing geometric loot shapes (gems, chests, ingots).

### 7. Hunter Size (Risk Mechanic)

- **7.1 [REFINED]** The Hunter initializes at session launch with a baseline length of 1 grid unit (head segment only).
- **7.2 [REFINED]** Segment increase rules:
  | Tier   | Target Value Range | Size Increase (Segments Added) |
  |--------|--------------------|-------------------------------|
  | Low    | 0 – 39             | +1                            |
  | Medium | 40 – 69            | +2                            |
  | High   | 70 – 89            | +3                            |
  | Elite  | 90 – 100           | +4                            |
- **7.3 [REFINED]** Size never decreases during a session. Every added segment expands physical collision footprint.

### 8. Scoring

- **8.1 [REFINED]** Total final session score is the cumulative sum of all completed level scores.
- **8.2 [REFINED]** Scoring formulas:
  ```
  level_time_bonus = max(0, (Max_Level_Time − level_time_seconds))
  level_score      = (level_target_value_sum × value_weight) + (level_time_bonus × time_weight)
  final_score      = Σ level_score_i  [for all completed levels]
  ```
- **8.3 [REFINED]** Config parameters: `value_weight` = 0.6, `time_weight` = 0.4, `Max_Level_Time` = 300 seconds.

### 9. Play Mode Selection

- **9.1 [REFINED]** On startup screen, players can select between:
  - Mode 1: Viper Bounty Hunter (Capture criminal targets using CSV/JSON registry)
  - Mode 2: Treasure Vault (Extract glowing vector loot gems & chests)
- **9.2 [REFINED]** Selection dynamically updates target visuals, score rules, and UI overlay themes.

### 10. Mobile Touch Controls & Responsiveness

- **10.1 [REFINED]** Mobile viewports render an interactive Virtual D-Pad overlay with Up/Down/Left/Right touch buttons.
- **10.2 [REFINED]** Touch inputs feed directly into `InputHandler.injectDirection()` with full 180-degree reversal validation.
- **10.3 [REFINED]** Responsive CSS styling prevents top viewport truncation on mobile screens.

### 11. Roaming Criminal Big Boss Hazard

- **11.1 [REFINED]** On specific level triggers, a roaming Big Boss figure hazard spawns on the grid matrix.
- **11.2 [REFINED]** The Big Boss figure moves autonomously on discrete grid cells.
- **11.3 [REFINED]** Colliding with the Big Boss figure triggers an instant Game Over state.

### 12. Cyberpunk Visual Experience & Particle FX

- **12.1 [REFINED]** Canvas rendering engine features neon glowing stroke styling, dark cyberpunk color palettes, and glowing grid lines.
- **12.2 [REFINED]** Capturing targets triggers dynamic particle explosion animations rendered over the grid matrix.
