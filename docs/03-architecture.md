## Component map

### RegistryService
- **Responsibility:** Fetches and parses criminal registry data or treasure vault loot definitions; provides play mode state management.
- **Exposes:** `loadRegistry()`, `getUnspawnedRecords()`, `getPlayMode()`, `setPlayMode(mode)`
- **Depends on:** Google Sheets CSV endpoint, Local filesystem (`data/fallback_registry.json`)
- **Data owned:** Target Record Pool (source of truth for active play mode target collection)
- **Technology:** JavaScript Fetch API — Native, lightweight, no third-party libraries needed.
- **Scaling strategy:** Stateless client-side fetch.
- **Failure mode:** Graceful degradation — Falls back to local `fallback_registry.json` on network failure or 5s timeout.

### FirebaseService
- **Responsibility:** Handles cloud-based persistence of player profiles and high scores.
- **Exposes:** `getProfiles()`, `saveProfile(name)`, `updateHighScore(name, score)`, `getGameRules()`
- **Depends on:** Firebase SDK (dynamic CDN import), Firestore Database, Local Storage (fallback)
- **Data owned:** Player profiles, session high scores, and game configuration rules.
- **Technology:** Firebase JS Firestore SDK — Dynamic module imports to prevent module load failures when offline.
- **Scaling strategy:** Firebase Firestore serverless database.
- **Failure mode:** Graceful degradation — Transparently falls back to browser `localStorage` if unconfigured or offline.

### GameLoop
- **Responsibility:** Orchestrates the fixed-interval tick rate and coordinates state updates across mode selection, movement, hazards, and collisions.
- **Exposes:** `start()`, `stop()`, `pause()`
- **Depends on:** InputHandler, GridState, CollisionDetector, Renderer, LevelManager
- **Data owned:** Tick timing, Active Session Status (Playing, GameOver), Play Mode configuration (`mode1` / `mode2`)
- **Technology:** `window.requestAnimationFrame` with delta time capping — Provides smooth execution tied to browser refresh rate.
- **Scaling strategy:** Stateless client-side execution.
- **Failure mode:** Hard failure — Halts game updates if exception occurs in tick cycle.

### InputHandler
- **Responsibility:** Captures keyboard events (WASD / Arrows) and virtual D-Pad touch events; validates direction changes.
- **Exposes:** `getCurrentDirection()`, `clearInputQueue()`, `injectDirection(direction)`
- **Depends on:** Browser DOM Events (keydown, touchstart/mousedown on Virtual D-Pad buttons)
- **Data owned:** Next intended Hunter direction, Input queue
- **Technology:** DOM Event Listeners & Virtual D-Pad touch routing — Supports both desktop and mobile web browsers.
- **Scaling strategy:** Stateless.
- **Failure mode:** Graceful degradation — Invalid keys or 180-degree reversal attempts are silently discarded.

### GridState
- **Responsibility:** Maintains spatial coordinates for Hunter, active Targets, Play Mode, and the roaming Big Boss hazard.
- **Exposes:** `moveHunter()`, `growHunter()`, `occupyCell()`, `getCellContents()`, `setBossPosition()`, `getBossPosition()`
- **Depends on:** None
- **Data owned:** Hunter body segments, Active Target coordinates, Play Mode (`mode1` / `mode2`), Big Boss coordinate & state
- **Technology:** In-memory spatial coordinate maps — Fast lookups for collision, hazard interaction, and rendering.
- **Scaling strategy:** Bounded by fixed grid size.
- **Failure mode:** Hard failure — Invalid coordinates throw exceptions.

### TargetManager
- **Responsibility:** Spawns targets onto unoccupied grid cells and prevents record repetition.
- **Exposes:** `spawnTarget()`, `handleCapture()`
- **Depends on:** GridState, RegistryService
- **Data owned:** Active target records currently on grid
- **Technology:** Pure JavaScript logic — Deterministic random cell selection with retry logic.
- **Scaling strategy:** CPU bound (retry algorithm for finding empty cells).
- **Failure mode:** Graceful degradation — If no cells are available, spawning is deferred to next tick.

### CollisionDetector
- **Responsibility:** Evaluates if the Hunter's head coordinate intersects a wall, self body segment, or roaming Big Boss figure.
- **Exposes:** `checkCollision(headCoord, gridBounds, bodySegments, bossCoord)`
- **Depends on:** None
- **Data owned:** None
- **Technology:** Pure JavaScript spatial logic — Coordinate equality checks across grid matrix bounds, self segments, and boss hazard coordinates.
- **Scaling strategy:** Stateless.
- **Failure mode:** Hard failure — Returns true unconditionally if data is malformed.

### ScoreManager
- **Responsibility:** Calculates target values, level time bonuses, and total session score.
- **Exposes:** `addCaptureValue()`, `completeLevel()`, `getSessionScore()`
- **Depends on:** None
- **Data owned:** Current level score, Level time elapsed, Final session score
- **Technology:** Pure JavaScript math logic — Implements defined scoring formulas with time bonus floor protection.
- **Scaling strategy:** Stateless.
- **Failure mode:** Graceful degradation — Floors negative time bonuses to zero to prevent score subtraction.

### LevelManager
- **Responsibility:** Controls level progression, hunter recentering, level target pools, and Big Boss hazard spawning.
- **Exposes:** `checkLevelCompletion()`, `advanceLevel()`, `spawnBossHazard()`
- **Depends on:** GridState, TargetManager, ScoreManager
- **Data owned:** Active level index, Captured target counter per level, Boss spawn state
- **Technology:** Pure JavaScript level state controller.
- **Scaling strategy:** Stateless client-side logic.
- **Failure mode:** Resets hunter safely to mathematical grid center on new level start.

### Renderer
- **Responsibility:** Renders the cyberpunk canvas grid matrix, neon hunter segments, targets/loot shapes, roaming Big Boss hazard, glowing UI indicators, and particle FX explosions.
- **Exposes:** `renderFrame(state)`, `triggerCaptureParticles(x, y)`
- **Depends on:** HTML5 Canvas 2D API, DOM elements (for UI overlays)
- **Data owned:** Image asset cache, particle explosion array
- **Technology:** HTML5 Canvas 2D API with vector neon shadows, dynamic canvas scaling, and particle physics animation loops.
- **Scaling strategy:** GPU/Browser bound.
- **Failure mode:** Graceful degradation — Renders vector loot shapes or placeholder images if avatar assets fail to load.

## API contracts

### GET [Google Sheets CSV Endpoint]
- **Auth:** None (Public read-only)
- **Request:** `GET` request to published Google Sheets URL
- **Response 200:** `text/csv` containing headers: `Name, Avatar_Asset_Path, Interpol_Red_Notice, FBI_Most_Wanted, Conviction_Status`
- **Response Network Error / Timeout (5s):** Triggers internal fallback mechanism.
- **Response Malformed (Missing headers):** Handled internally by skipping invalid records.
- **Idempotency:** Yes — Repeated calls yield the latest published sheet without side effects.

## NFR mapping

| NFR | Architectural decision |
|-----|----------------------|
| Seamless level progression | `LevelManager` instantly resets `GridState` (Hunter to center) and calls `TargetManager` to spawn new targets without pausing `GameLoop`. |
| Offline playability | `RegistryService` intercepts network failures and reads `fallback_registry.json` packaged within the build directory. |
| Security: XSS Prevention | `RegistryService` strips HTML tags from text fields; `Renderer` draws avatars via Canvas `drawImage` rather than DOM injection, mitigating script execution. |
| Time limit constraint | `ScoreManager` tracks delta time; applies `max(0, Max_Level_Time - elapsed)` to ensure no negative penalties. |
| Persistent Cloud Saves | `FirebaseService` handles cloud-based profile management, falling back to local `localStorage` if configuration credentials or network access is missing. |
| Responsive Mobile Controls | `InputHandler` and CSS3 touch overlays provide Virtual D-Pad controls with dynamic viewport scaling. |
| Cyberpunk Visual Performance | `Renderer` uses HTML5 Canvas vector shadow glow and particle arrays flushed per frame tick. |

## Risk flags

| Risk | Consequence | Mitigation |
|------|-------------|-----------|
| Third-party API with no circuit breaker or fallback (Google Sheets) | Game cannot start if sheet is deleted or private. | Fallback implemented via bundled `fallback_registry.json` loaded upon 5s timeout. |
| Single point of failure with no fallback | Corrupted `fallback_registry.json` combined with offline status. | Hard block on gameplay (intentional per US-002); notify player with clear error UI. |
| No auth boundary on an externally accessible interface | Google Sheet data could be scraped by unintended parties. | Acceptable risk: Public data is intended. Rate limiting relies on Google's infrastructure. |
| Compliance requirement with no corresponding design control | Real identities in public deployment leading to legal liability. | Manual CI/CD gate process (US-015); process-based mitigation outside system architecture. |
| Third-party Cloud persistence downtime/misconfiguration | App hangs or crashes during load or score updates due to database issues. | `FirebaseService` executes dynamic imports and validates credentials. Falling back to local storage handles server errors transparently. |

## Architecture Decision Records

### ADR-001 — Offline Registry Strategy
- **Status:** Accepted
- **Context:** The game must function offline, requiring criminal data even when the Google Sheet is unreachable.
- **Options considered:**
  - Option A: Service Worker caching of the CSV endpoint — Pro: Transparent to app code. Con: Fails on very first launch if offline; requires HTTPS setup.
  - Option B: Bundled JSON file fallback — Pro: Guaranteed to exist on disk; works on first launch. Con: Static data may become stale.
- **Decision:** Option B (Bundled JSON file). It guarantees the application is never bricked purely due to network conditions on first load.
- **Consequences:** Easier to test offline mode locally. Stale data in fallback is accepted as a trade-off for reliability.

### ADR-002 — Rendering Engine Choice
- **Status:** Accepted
- **Context:** The grid and entities need to be drawn efficiently and refreshed on a fixed tick rate.
- **Options considered:**
  - Option A: DOM Manipulation (div grid) — Pro: Simple CSS styling. Con: Poor performance for rapid reflows; harder to prevent XSS.
  - Option B: HTML5 Canvas 2D — Pro: Excellent performance for tile games; inherently safer against XSS since assets are drawn as pixels.
- **Decision:** Option B (HTML5 Canvas 2D). It provides the performance ceiling necessary for a smooth game loop and simplifies the security profile.
- **Consequences:** UI styling requires drawing text/shapes manually rather than using CSS.

### ADR-003 — Asset Loading Security
- **Status:** Accepted
- **Context:** The CSV provides `Avatar_Asset_Path` which could contain malicious URIs.
- **Options considered:**
  - Option A: Pre-download all assets at build time — Pro: 100% secure. Con: Dynamic sheet updates are impossible without a new build.
  - Option B: Dynamic loading with Canvas rendering — Pro: Supports dynamic URLs. Con: Relies on browser CORS and Canvas security.
- **Decision:** Option B. Image loading will be restricted to `Image` objects drawn onto Canvas. Any non-image data (like `javascript:`) will naturally fail to render as an image, providing implicit sanitisation.
- **Consequences:** We accept CORS errors for improperly configured external image hosts; placeholders will be used as the graceful degradation.

### ADR-004 — Cloud Database Persistence with Local-First Fallback
- **Status:** Accepted
- **Context:** Player profile high scores were previously ephemeral, stored only in browser cache (`localStorage`).
- **Options considered:**
  - Option A: Local Node.js Express server + SQLite — Pro: Self-contained. Con: Requires separate backend runtime.
  - Option B: Firebase Firestore (BaaS) — Pro: Native client-side API, serverless scalability. Con: External credential key dependency.
- **Decision:** Option B (Firebase Firestore). If Firebase config is missing or network fails, engine reverts to browser `localStorage` transparently.
- **Consequences:** Provides persistent multi-device capability without losing offline availability.

### ADR-005 — Dual Play Modes Architecture
- **Status:** Accepted
- **Context:** Players need distinct gameplay experiences: Bounty Hunter target pursuit (`mode1`) and Treasure Vault loot extraction (`mode2`).
- **Decision:** Encapsulate mode selection in `RegistryService` and propagate `playMode` context through `GameLoop`, `GridState`, and `Renderer`. `mode1` renders avatar target cards; `mode2` renders vector loot shapes (gems, chests, ingots).
- **Consequences:** Clean separation of mode concerns without duplicating game loop or collision physics logic.

### ADR-006 — Mobile D-Pad Touch Controls Overlay
- **Status:** Accepted
- **Context:** Mobile web users require touch controls without obstructing canvas viewports or causing top/bottom cutoffs.
- **Decision:** Implement a CSS flex/grid virtual D-Pad overlay and expose `injectDirection()` in `InputHandler` to seamlessly enqueue touch directional changes into the same tick queue used by keyboard events.
- **Consequences:** Identical input validation and 180-degree reversal protection apply universally across mobile and desktop.

### ADR-007 — Roaming Criminal Big Boss Figure Hazard
- **Status:** Accepted
- **Context:** Higher gameplay difficulty requires dynamic moving hazards on the grid.
- **Decision:** Add autonomous boss position state to `GridState` and evaluate boss collisions in `CollisionDetector`. `LevelManager` triggers boss spawning based on level progression thresholds, and `Renderer` handles glowing boss graphics.
- **Consequences:** Adds strategic depth to gameplay while reusing spatial collision physics.
