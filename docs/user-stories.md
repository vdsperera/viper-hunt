## US-001

**Title:** Load criminal registry at launch

**Statement:**  
As a **player**, I want the game to load the criminal registry automatically when I open it, so that target data is ready before gameplay begins without any manual action on my part.

**Priority:** Must have  
_Rationale: The entire target pool depends on registry data; nothing can be displayed or played without it._

**Assumptions:**
- The Google Sheet is published as a publicly readable CSV export endpoint.
- The player's browser has internet access at launch time.
- A valid `fallback_registry.json` is packaged in the build directory.

**Out of scope:**
- Authenticating the player or admin to fetch the sheet.
- Allowing the player to switch or select alternative registries.
- Live hot-reloading of registry data during an active session.
- Any admin CRUD operations on the sheet (see US-014).

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: successful remote fetch
Given the player opens the game in a desktop browser
And the Google Sheets CSV endpoint is reachable
When the application initialises
Then the system fetches the registry within 5 seconds
And all records are parsed and loaded into the active session pool
And the game Start screen becomes interactive

# Scenario 2 — Sad path: network timeout
Given the player opens the game
And the Google Sheets endpoint does not respond within 5 seconds
When the fetch timeout elapses
Then the system silently loads the bundled fallback_registry.json
And the player is not shown an error dialog
And the game Start screen becomes interactive

# Scenario 3 — Sad path: malformed CSV response
Given the Google Sheets endpoint responds
And the CSV payload is missing one or more required field headers (Name, Avatar_Asset_Path, Interpol_Red_Notice, FBI_Most_Wanted, Conviction_Status)
When the system attempts to parse the response
Then the malformed records are skipped with an internal warning logged to the browser console
And only valid records are loaded into the session pool
And if zero valid records remain, the system falls back to fallback_registry.json

# Scenario 4 — Edge case: registry contains exactly 500 records
Given the CSV export contains exactly 500 records, all valid
When the application initialises
Then all 500 records are loaded into the session pool with no truncation

# Scenario 5 — Security: unauthenticated access to raw sheet
Given the Google Sheet is configured with public read-only access via CSV export URL
When the client fetches the endpoint
Then no OAuth token or session credential is transmitted or stored by the client
And no write permissions are exercised by the client fetch operation
```

## US-002

**Title:** Fall back to local registry when offline

**Statement:**  
As a **player**, I want the game to use a locally bundled criminal registry when my internet connection is unavailable, so that I can play without requiring network access.

**Priority:** Must have  
_Rationale: Offline play is an explicit constraint in §1.3; blocking the game on connectivity failure is unacceptable._

**Assumptions:**
- `fallback_registry.json` is pre-packaged in the game build and passes the same structural validation as the remote CSV.
- The offline check occurs at the application launch fetch attempt, not mid-session.

**Out of scope:**
- Syncing or updating the local fallback file during gameplay.
- Notifying the player that offline mode is active.
- Admin ability to edit the fallback file at runtime.

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: offline fallback loads successfully
Given the player opens the game
And the system is offline or the fetch times out
When fallback_registry.json is read from the build directory
Then all valid records are loaded into the session pool
And the game Start screen becomes interactive

# Scenario 2 — Sad path: fallback file is missing or corrupt
Given the player opens the game
And the network is unavailable
And fallback_registry.json is absent or is not valid JSON
When the system attempts to load the fallback
Then the game displays a friendly error screen stating that the registry cannot be loaded
And the player is not permitted to start a gameplay session

# Scenario 3 — Edge case: fallback file contains zero valid records
Given fallback_registry.json is valid JSON but contains an empty array
When the system loads it
Then the game displays an error screen stating the registry is empty
And the player cannot start a session

# Scenario 4 — Security: fallback file is read-only at runtime
Given the game is running
When any game module reads fallback_registry.json
Then the file is opened in read-only mode
And no write, append, or delete operation is performed on the file during the session
```

## US-003

**Title:** Control the Hunter with keyboard input

**Statement:**  
As a **player**, I want to change the Hunter's movement direction using the Arrow Keys or WASD keys, so that I can navigate the grid and steer toward targets.

**Priority:** Must have  
_Rationale: Directional control is the sole interaction mechanism for the core game loop._

**Assumptions:**
- The game canvas has browser keyboard focus when the session is active.
- The grid tick rate is defined and constant (frame-based).
- The Hunter always moves forward automatically; the player only changes direction.

**Out of scope:**
- Mouse, touch, or gamepad input controls.
- Pausing the game via keyboard (separate concern).
- Changing movement speed through input.

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: valid direction change
Given the Hunter is moving Right
And the player presses the Up Arrow key (or W key)
When the next game tick processes
Then the Hunter's head advances one grid cell Upward
And all trailing body segments shift to follow the previous head position

# Scenario 2 — Happy path: WASD alternative input
Given the Hunter is moving Up
And the player presses the D key
When the next game tick processes
Then the Hunter moves Right

# Scenario 3 — Sad path: forbidden 180-degree reversal
Given the Hunter is moving Right and has at least 2 body segments
When the player presses the Left Arrow key (or A key)
Then the input is discarded
And the Hunter continues moving Right on the next tick

# Scenario 4 — Edge case: multiple keys pressed within a single tick
Given the Hunter is moving Up
And the player presses Right and then Down in rapid succession within one tick interval
When the tick processes
Then only the first valid direction input (Right) is applied
And the second input (Down) is queued or discarded, not causing an instant 180-degree turn

# Scenario 5 — Edge case: Hunter has length 1 (head only) and player attempts reversal
Given the Hunter is length 1 (head only)
And the Hunter is moving Right
When the player presses the Left Arrow key
Then the input is discarded
And the Hunter continues moving Right
```

## US-004

**Title:** Detect and trigger game over on collision

**Statement:**  
As a **player**, I want the game to immediately end my session when the Hunter hits a wall or its own body, so that self-navigation risk is the central challenge of gameplay.

**Priority:** Must have  
_Rationale: Collision detection is the primary failure condition that gives the game its tension._

**Assumptions:**
- Grid boundaries are fixed at 1280×720 canvas resolution with a defined cell size.
- Self-collision check compares the incoming head coordinate against all current trailing segment coordinates.
- Game Over is instantaneous on the frame the collision is detected.

**Out of scope:**
- Shield or invincibility power-ups (explicitly excluded in §9.1).
- Wrap-around screen mechanics (explicitly excluded in §4.4).
- Saving the partial score before Game Over (see US-012).

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: boundary collision triggers game over
Given the Hunter is moving Right
And the Hunter's head is at the rightmost column of the grid
When the next game tick processes
Then the Hunter's head coordinate exceeds the grid boundary
And the Game Over sequence is triggered instantly

# Scenario 2 — Happy path: self-collision triggers game over
Given the Hunter has a body length of 5 or more segments
And the Hunter's head is adjacent to one of its own trailing body segments
When the player steers the Hunter's head onto that body segment's coordinate
Then the Game Over sequence is triggered instantly

# Scenario 3 — Sad path: no collision on a valid open cell move
Given the Hunter is moving in any cardinal direction
And the next cell in that direction is within bounds and unoccupied by the Hunter's body
When the next tick processes
Then the Hunter advances without triggering game over

# Scenario 4 — Edge case: head approaches but does not enter collision cell
Given the Hunter's head is two cells away from the boundary
When the player does not steer into the boundary
Then no game over is triggered

# Scenario 5 — Security: game state cannot be externally manipulated to skip collision
Given an active gameplay session
When a browser console script attempts to alter the Hunter's position or collision flag
Then the internal game loop re-evaluates collision on every tick regardless of external state tampering
```

## US-005

**Title:** Capture a target by overlapping its cell

**Statement:**  
As a **player**, I want the Hunter to automatically capture a target when its head enters that target's grid cell, so that I can score points through skillful navigation.

**Priority:** Must have  
_Rationale: Target capture is the primary scoring action and core gameplay reward loop._

**Assumptions:**
- Targets occupy exactly one grid cell.
- Capture detection runs every game tick by comparing the Hunter's head coordinate against all active target coordinates.
- The captured target's record ID is removed from the session pool immediately.

**Out of scope:**
- Mouse-click or tap interaction with targets (explicitly excluded in §6.3).
- Capturing targets with body segments (only the head triggers capture).
- Multiple captures in a single tick (one capture event per tick per head position).

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: head enters target cell
Given a target is active at grid coordinate (10, 5)
And the Hunter's head is at (9, 5) moving Right
When the next game tick processes
Then the Hunter's head moves to (10, 5)
And the capture event fires: the target is removed from the grid
And the player's score is updated with the target's value
And a new target spawns if remaining unspawned targets exist

# Scenario 2 — Sad path: body segment passes through target cell (no capture)
Given a target is at (10, 5)
And the Hunter's head is at (11, 5) moving Left
But a body segment (not head) is currently at (10, 5) due to prior movement
Then no capture event fires for that tick (body segments do not trigger capture)

# Scenario 3 — Edge case: last target of a level is captured
Given the level has 1 remaining active target
And the Hunter captures it
Then the level complete event fires
And the next level begins immediately (no splash screen delay per §5.1)

# Scenario 4 — Edge case: target spawns on the Hunter's body cell
Given all unoccupied grid cells are zero
When a new target spawn is triggered
Then the spawn algorithm retries until an unoccupied cell is found
And no target is placed on a cell currently occupied by any Hunter segment
```

## US-006

**Title:** Grow the Hunter after a capture

**Statement:**  
As a **player**, I want the Hunter to grow longer after capturing a target, so that I face increasing navigational difficulty that reflects my score progression.

**Priority:** Must have  
_Rationale: Growth is the core risk mechanic that makes higher-value targets more consequential._

**Assumptions:**
- Growth segments are appended to the tail, not the head.
- Segment addition happens on the frame tick following the capture event.
- The value tier table (§7.2) is the authoritative mapping.

**Out of scope:**
- Reducing Hunter length (no shrink mechanic in MVP, §7.3).
- Visual distinction between segments added from different captures.
- A cap on maximum Hunter length (§7.3 explicitly excludes a cap).

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: Low-tier target captured (value 0–39)
Given the Hunter has a current length of 3 segments
And the player captures a target with a computed value of 25
When the capture event is processed
Then 1 segment is appended to the Hunter's tail on the next tick
And the Hunter's total length becomes 4

# Scenario 2 — Happy path: Elite-tier target captured (value 90–100)
Given the Hunter has a current length of 3 segments
And the player captures a target with a computed value of 100
When the capture event is processed
Then 4 segments are appended to the Hunter's tail
And the Hunter's total length becomes 7

# Scenario 3 — Edge case: target with value exactly 40 (Medium tier boundary)
Given the player captures a target with a computed value of 40
When the capture event is processed
Then 2 segments are appended (Medium tier applies at value ≥ 40)

# Scenario 4 — Edge case: target with value exactly 70 (High tier boundary)
Given the player captures a target with a computed value of 70
When the capture event is processed
Then 3 segments are appended (High tier applies at value ≥ 70)

# Scenario 5 — Edge case: target with value 0 (no flags set, Low tier)
Given a target record where all boolean flags are 0
And its computed value is 0
When the player captures it
Then 1 segment is appended (Low tier: 0–39)
```

## US-007

**Title:** Progress to the next level automatically

**Statement:**  
As a **player**, I want the game to advance me to the next level the instant I capture all targets in the current level, so that momentum is maintained without interruption.

**Priority:** Must have  
_Rationale: Seamless level progression is an explicit design constraint in §5.1._

**Assumptions:**
- Level completion is determined by: captured_count == total_level_targets (N).
- Level counter increments by 1 on completion.
- Level config parameters (`total_level_targets`, `max_simultaneous_targets`) are pre-defined externally.

**Out of scope:**
- Displaying an interim splash screen or level summary (explicitly excluded in §5.1).
- Allowing the player to skip or replay levels.
- Resetting the Hunter's length on level transition (§5.2 explicitly excludes this).

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: final target of a level captured
Given the player is on Level 1 with total_level_targets = 10
And the player has captured 9 targets
When the player captures the 10th target
Then the level counter increments to Level 2
And new targets for Level 2 are rendered on the grid within the same frame
And no pause, animation, or splash screen is displayed

# Scenario 2 — Sad path: level does not advance prematurely
Given the player is on Level 1 with total_level_targets = 10
And the player has captured 9 targets
When 1 target remains active on the grid
Then the level counter remains at Level 1
And no new level targets are spawned

# Scenario 3 — Edge case: Level 1 configured with total_level_targets = 1
Given total_level_targets for Level 1 is 1
And the player captures the single target
Then Level 2 begins immediately
```

## US-008

**Title:** Reset Hunter position at level start

**Statement:**  
As a **player**, I want the Hunter to reposition to the centre of the grid at the start of each new level, so that I begin each level from a predictable, safe starting position regardless of where I ended the previous level.

**Priority:** Must have  
_Rationale: Without repositioning, carry-over head positions may cause immediate collisions on the new level's grid (§5.3, REQ-005)._

**Assumptions:**
- "Centre grid coordinate" is defined as floor(X_cells / 2), floor(Y_cells / 2).
- Body segments trail horizontally to the left of the head.
- Wrap-around along the grid perimeter is only invoked if length exceeds available left-side space.

**Out of scope:**
- Resetting the Hunter's length (maintained across levels, §5.2).
- Allowing the player to choose the starting position.
- Changing orientation direction at start (always faces Right by default unless specified otherwise).

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: short Hunter repositions cleanly
Given the player completes Level 1 with a Hunter of length 3
When Level 2 begins
Then the Hunter's head is placed at the centre grid coordinate
And the 2 trailing segments extend horizontally to the left without overlap
And no self-collision is detected at the start frame

# Scenario 2 — Edge case: Hunter length fills entire row width
Given the grid has 40 columns
And the Hunter has a length of 40 segments at level start
When the Hunter is repositioned
Then the head is at the centre column
And tail segments fill leftward to column 0, then wrap along the grid perimeter
And no segment occupies the head's starting coordinate

# Scenario 3 — Edge case: Hunter length of 1 (head only)
Given the Hunter has a length of 1 at level start
When repositioned
Then the head is placed at the centre coordinate with no trailing segments

# Scenario 4 — Sad path: centre coordinate is occupied by a freshly spawned target
Given a new level begins
And a target happens to spawn at the exact centre coordinate before the Hunter is placed
Then the spawn algorithm is re-run to relocate that target to another unoccupied cell
And the Hunter head takes the centre coordinate without a forced capture event on frame 0
```

## US-009

**Title:** Spawn targets within level constraints

**Statement:**  
As a **player**, I want targets to appear on the grid according to the level's simultaneous target limit, so that each level presents a distinct density challenge.

**Priority:** Must have  
_Rationale: Target spawning logic is central to level difficulty design (§5.4, REQ-004)._

**Assumptions:**
- `max_simultaneous_targets` ≤ `total_level_targets` (N) for any given level config.
- A new target spawns instantly the moment a capture reduces the active count below the limit and unspawned targets remain.
- Targets do not despawn; they persist until captured (§5.5).

**Out of scope:**
- Targets with timers or expiry (explicitly excluded in §5.5).
- Player control over spawn positions.
- Spawning more targets than N for a level.

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: initial spawn fills simultaneous limit
Given Level 1 has total_level_targets = 10 and max_simultaneous_targets = 3
When Level 1 begins
Then exactly 3 targets are rendered on random unoccupied grid cells

# Scenario 2 — Happy path: capture triggers immediate respawn
Given 3 targets are active and the player captures 1
And 7 unspawned targets remain in the level pool
When the capture event fires
Then 1 new target spawns immediately on a random unoccupied cell
And the active count returns to 3

# Scenario 3 — Edge case: remaining pool drops below simultaneous limit
Given total_level_targets = 5 and max_simultaneous_targets = 3
And 4 targets have already been captured
And 1 unspawned target remains
When the player captures 1 of the 2 currently active targets
Then exactly 1 new target spawns (the last from the pool)
And active count becomes 2 (not 3)

# Scenario 4 — Edge case: no unoccupied cells available for spawn
Given the grid is near-full and a new target must spawn
When the spawn algorithm runs
Then it retries across all grid cells to find an unoccupied one
And no target is placed on a cell occupied by any Hunter segment or existing target

# Scenario 5 — Sad path: level config has max_simultaneous_targets = 0
Given a level is configured with max_simultaneous_targets = 0
When the level begins
Then zero targets are rendered
And the level immediately completes if total_level_targets = 0, otherwise the game logs a config error and halts
```

## US-010

**Title:** Display target avatar and point value on-grid

**Statement:**  
As a **player**, I want each target to show its avatar image and numeric point value directly on the grid cell, so that I can make informed decisions about which targets to pursue based on their risk-reward ratio.

**Priority:** Must have  
_Rationale: The avatar and value are the primary in-game information signals for player decision-making (§6.2)._

**Assumptions:**
- Avatar assets are reachable via the `Avatar_Asset_Path` field (URL or local path).
- The numeric value is the pre-calculated integer from the §6.1 formula.
- Rendering occurs when the target spawns and persists until capture.

**Out of scope:**
- Tooltip, hover, click, or tap interactions on targets (explicitly excluded in §6.3).
- Displaying the criminal's `Name` field on the grid (not specified).
- Animating or pulsing target icons.

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: target renders with avatar and value
Given a target is spawned with Avatar_Asset_Path = "https://cdn.example.com/avatar1.png" and a computed value of 75
When the target is placed on the grid
Then the avatar image is rendered within the target's grid cell bounds
And the integer "75" is displayed adjacent to the avatar icon

# Scenario 2 — Sad path: avatar image URL is unreachable
Given a target's Avatar_Asset_Path URL returns a 404 or network error
When the target is rendered
Then a placeholder avatar graphic is displayed in place of the missing image
And the numeric value is still displayed correctly

# Scenario 3 — Edge case: target value is 0 (all flags are 0)
Given a target record with all boolean fields set to 0
And its computed value is 0
When the target renders
Then "0" is displayed as the numeric value alongside the placeholder or avatar

# Scenario 4 — Edge case: target value is 100 (all flags set, maximum)
Given Interpol_Red_Notice = 1, FBI_Most_Wanted = 1, Conviction_Status = 1
And computed value = 40 + 35 + 25 = 100
When the target renders
Then "100" is displayed alongside the avatar

# Scenario 5 — Security: avatar path cannot inject executable content
Given a target record's Avatar_Asset_Path contains a non-image URI (e.g., javascript: or data:text/html)
When the system attempts to load the avatar
```

## US-011

**Title:** Calculate the level score with time bonus

**Statement:**  
As a **player**, I want my score for each level to reflect both the total value of targets I captured and how quickly I completed the level, so that skilled and fast play is rewarded.

**Priority:** Must have  
_Rationale: The scoring formula is the quantitative feedback loop that drives player engagement (§8.3)._

**Assumptions:**
- The level timer starts at 0.0 when the level begins and stops at the capture of the final target.
- `Max_Level_Time` default is 300 seconds (configurable).
- `value_weight` = 0.6, `time_weight` = 0.4 (configurable).
- `level_target_value_sum` is the sum of all individual captured target values within the level.

**Out of scope:**
- Displaying a breakdown of value vs. time bonus to the player (not specified).
- Penalising the player for exceeding Max_Level_Time (score floors at zero for time component via `max(0, …)`).
- Modifying weights mid-session.

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: level completed well within time limit
Given a level where level_target_value_sum = 200 and level_time_seconds = 60
And Max_Level_Time = 300, value_weight = 0.6, time_weight = 0.4
When the level is completed
Then level_time_bonus = max(0, 300 − 60) = 240
And level_score = (200 × 0.6) + (240 × 0.4) = 120 + 96 = 216

# Scenario 2 — Edge case: level completed exactly at Max_Level_Time
Given level_time_seconds = 300 and Max_Level_Time = 300
When the level is completed
Then level_time_bonus = max(0, 300 − 300) = 0
And level_score = (level_target_value_sum × 0.6) + 0

# Scenario 3 — Edge case: level completed in near-zero time (< 1 second)
Given level_time_seconds = 0.001
When the level is completed
Then level_time_bonus = max(0, 300 − 0.001) ≈ 299.999
And no division-by-zero error occurs
And the score is calculated correctly using the formula

# Scenario 4 — Edge case: level time exceeds Max_Level_Time
Given level_time_seconds = 400 and Max_Level_Time = 300
When the level is completed
Then level_time_bonus = max(0, 300 − 400) = 0
And the time component does not produce a negative score contribution

# Scenario 5 — Sad path: all captured targets have value 0
Given every target in the level has computed value 0
And level_target_value_sum = 0
When the level is completed quickly at 10 seconds
Then level_score = (0 × 0.6) + (290 × 0.4) = 116
And a non-zero score is still awarded for speed
```

## US-012

**Title:** Accumulate the final session score

**Statement:**  
As a **player**, I want my final score to be the sum of all level scores when my session ends, so that every level I complete contributes meaningfully to my overall result.

**Priority:** Must have  
_Rationale: The cumulative final score is the primary performance metric of a session (§8.1)._

**Assumptions:**
- `final_score = Σ level_score_i` for all fully completed levels at the point of Game Over.
- A level is "completed" only when all N targets are captured; a level in progress at Game Over contributes 0.
- The score persists in memory throughout the session.

**Out of scope:**
- Persisting the score to a leaderboard or remote database (not specified for MVP).
- Resetting the score mid-session without a full session restart.
- Displaying granular per-level breakdowns on the Game Over screen (not specified).

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: multiple levels completed before game over
Given the player completes Level 1 (level_score = 216) and Level 2 (level_score = 180)
And the player collides on Level 3
When the Game Over screen is displayed
Then final_score = 216 + 180 = 396

# Scenario 2 — Edge case: game over on Level 1 before capturing any target
Given the player dies at Level 1 without capturing any target
When Game Over triggers
Then final_score = 0

# Scenario 3 — Edge case: game over mid-level after some captures
Given the player completes Level 1 (level_score = 150) and then captures 2 targets in Level 2 before dying
When Game Over triggers
Then final_score = 150
And Level 2's partial captures do not contribute to final_score

# Scenario 4 — Sad path: score variable is read before any level completes
Given the player triggers Game Over on the very first tick of Level 1
When the Game Over screen renders
Then the final_score displays as 0 without error
```

## US-013

**Title:** Restrict each criminal record to one capture per session

**Statement:**  
As a **player**, I want each criminal record to appear at most once during my entire gameplay session, so that the registry functions as a finite, non-repeating pool and capturing rare targets feels meaningful.

**Priority:** Should have  
_Rationale: Prevents record repetition, preserving the thematic integrity of the criminal-hunt narrative (§3.4)._

**Assumptions:**
- A session is defined as the period from the initial registry load until Game Over or browser tab close.
- The session pool is an in-memory exclusion set of consumed record IDs.
- The pool is never written to or persisted between sessions.

**Out of scope:**
- Cross-session deduplication or persistent capture history.
- Notifying the player when the session pool is exhausted.
- Replenishing the pool mid-session.

**Acceptance Criteria:**

```gherkin
# Scenario 1 — Happy path: captured record is removed from pool
Given criminal record ID "C-042" is in the active session pool
And it spawns as a target and is captured by the player
When the capture event fires
Then record ID "C-042" is removed from the session spawn pool
And it cannot spawn again in any subsequent level of this session

# Scenario 2 — Edge case: all records in the session pool have been captured
Given a session where the total captured records equals the total registry size
When the system attempts to spawn a new target
Then no new target spawns
And the current level can still be completed if all configured targets have already been dispensed

# Scenario 3 — Sad path: spawn algorithm selects an already-captured record ID
Given the pool correctly excludes captured IDs
When the spawn algorithm runs
Then only records from the remaining unspawned pool are selected
And no previously captured record ID is chosen

# Scenario 4 — Security: session pool cannot be reset by browser refresh state manipulation
Given a session is active and 50 records have been captured
When the session data is queried externally (e.g., via browser console)
Then the exclusion set cannot be cleared without a full application reload
And the records remain excluded for the remainder of the current session
```

## US-014

**Title:** Admin updates registry between sessions

**Statement:**  
As a **system admin**, I want my changes to the Google Sheet to take effect the next time a player launches the game, so that I can manage the criminal registry without disrupting active gameplay sessions.

**Priority:** Should have  
_Rationale: Admin configuration is an explicit operational requirement for maintaining the target database (§2.2, §3.5)._

**Assumptions:**
- The admin has edit access to the designated Google Sheet.
- Changes include adding, modifying, or deleting rows.
- The game re-fetches the sheet from scratch on every new application launch.

**Out of scope:**
- Admin authentication within the game client.
- Live reloading of registry changes into an active session (explicitly excluded in §3.5).

- The legal evaluation covers defamation risk, right-of-publicity, and app storefront terms.

- Defining specific fictional asset creation workflows.
- Content moderation tooling within the game client.

```gherkin
When the admin initiates the public deployment pipeline

When the build is submitted to the compliance gate check
Then the gate flags the build as non-compliant
And the build is not cleared for public deployment
```
