### REQ-006
- **Type:** Conflict
- **Location:** Section 8: "✦ Time is measured per level... level_score = level_target_value_sum × value_weight + (1 / level_time_seconds) × time_weight"
- **Problem:** If a player completes a level in 0 seconds due to a frame glitch or edge execution, a division-by-zero error occurs (1/0). Furthermore, using linear reciprocal scaling means a difference between 1 and 2 seconds alters the score drastically, while the difference between 60 and 61 seconds is practically non-existent.
- **Question:** Can we implement a protected baseline duration or a standard linear time bonus subtraction model to prevent division-by-zero errors and normalize scoring?
- **Suggested fix:** Change to: "Level time tracking starts at 0.0 seconds and increments continuously. To prevent division-by-zero errors and balance scoring distributions, the time score component uses a linear countdown bonus: time_score = max(0, (Max_Level_Time − level_time_seconds)) × time_weight, where Max_Level_Time is a configurable integer (default: 300 seconds)."

## Refined Requirements

### 1. Platform

- **1.1 [REFINED]** MVP target platform is explicitly a web browser application built using standard HTML5 Canvas and JavaScript (ES6+), optimized for desktop web browsers.
- **1.2 [REFINED]** The application layout uses a fixed canvas rendering resolution of 1280×720 pixels (16:9 aspect ratio).
- **1.3 [REFINED]** Offline play is supported via client-side caching of local code assets and an integrated local fallback database.

### 2. Actors & Roles

- **2.1 [REFINED]** **Player:** An end-user who interacts with the game via keyboard inputs to control the direction of the Hunter through a 2D grid matrix.
- **2.2 [REFINED]** **System Admin:** A privileged user who modifies, appends, or deletes rows within a designated remote configuration spreadsheet (Google Sheets) to adjust game parameters and target definitions.

### 3. Criminal Registry

- **3.1 [REFINED]** The canonical target database is maintained via a Google Sheet during the MVP phase. At application launch, the client fetches runtime target definitions from this sheet. If the connection fails or if the system is offline, the app falls back to a locally bundled `fallback_registry.json` asset.
- **3.2 [REFINED]** For the private internal MVP lifecycle, the registry is pre-seeded with a minimum of 500 records containing synthetic data or real public record reference items. Real photos may be used for internal evaluations only. Any public distribution requires a complete swap to illustrated fictional avatar graphic assets.
- **3.3 [REFINED]** Each record must strictly contain the following data fields:
  - `Name` (String)
  - `Avatar_Asset_Path` (String URL or local path)
  - `Interpol_Red_Notice` (Boolean: 0 or 1)
  - `FBI_Most_Wanted` (Boolean: 0 or 1)
  - `Conviction_Status` (Boolean: 0 or 1)
- **3.4 [REFINED]** A specific criminal record can spawn at most once per single gameplay session. Once captured by the Hunter, that specific record ID is removed from the active session pool.
- **3.5 [REFINED]** Modifications made to the Google Sheet by the Admin do not inject live hot-reloads into active gameplay sessions. Changes apply dynamically at the next application initialization or session reset.

### 4. Core Game Mechanic

- **4.1 [REFINED]** The game area operates on a discrete grid pattern of X × Y cells. The Hunter occupies a sequential chain of grid cells and automatically advances forward by 1 grid unit per game frame tick in the last registered cardinal direction (Up, Down, Left, Right).
- **4.2 [REFINED]** A gameplay session terminates instantly, triggering the Game Over sequence, if the head coordinate of the Hunter matches any coordinate of its own trailing body segments (self-collision) or matches any coordinate outside the defined grid bounds (boundary collision).
- **4.3 [REFINED]** Overlapping the Hunter's head coordinate with a target's coordinate triggers a capture event: the target is deleted, the score is updated, and the Hunter's body segments grow based on the target value tier.
- **4.4 [REFINED]** No additional mechanics (such as speed changes, shooting, wrap-around screens, or obstacles) are included in this MVP scope.

### 5. Level Structure

- **5.1 [REFINED]** Game progression is broken down into consecutive sequential levels. The moment the number of captured targets equals the total targets configured for the active level (N), the system increments the level counter and instantly renders the new level's targets without pausing or displaying interim splash screens.
- **5.2 [REFINED]** The Hunter's size (total number of trailing segments) does not reset when transitioning to a new level; the full structural length is maintained throughout the entire session.
- **5.3 [REFINED]** At the precise start frame of a new level, the Hunter's head coordinate is moved to the exact center grid coordinate. Trailing body segments are drawn extending horizontally to the left. If the length exceeds available grid space, the tail wraps linearly along the grid perimeter without colliding.
- **5.4 [REFINED]** Levels are configured externally via parameters defining:
  - `total_level_targets` (N)
  - `max_simultaneous_targets` (Maximum active icons visible on-screen at one instant)
- **5.5 [REFINED]** Targets do not possess an active expiration or despawn timer; they remain anchored to their assigned grid coordinates until successfully captured or until the session terminates.

### 6. Target Value & Display

- **6.1 [REFINED]** The individual base value of a target record is dynamically calculated on ingestion using the following expression:

  ```
  target_value = (interpol_red_notice × 40) + (fbi_most_wanted × 35) + (conviction_status × 25)
  ```

- **6.2 [REFINED]** When a target is drawn on the grid, it displays its unique visual avatar graphic alongside a numeric integer value text block rendered directly adjacent to the icon bounds.
- **6.3 [REFINED]** Clicking, hovering, or tapping on target entities yields no interaction or operational behavior during the MVP phase.

### 7. Hunter Size (Risk Mechanic)

- **7.1 [REFINED]** The Hunter initializes at session launch with a baseline length of 1 grid unit (head segment only).
- **7.2 [REFINED]** Upon capturing a target, additional body segments are appended to the tail during the next frame tick according to the target's value tier:

  | Tier   | Target Value Range | Size Increase (Segments Added) |
  |--------|--------------------|-------------------------------|
  | Low    | 0 – 39             | +1                            |
  | Medium | 40 – 69            | +2                            |
  | High   | 70 – 89            | +3                            |
  | Elite  | 90 – 100           | +4                            |

- **7.3 [REFINED]** The Hunter's segment size never decreases during a session. There is no structural cap or upper limit enforced on maximum size for the MVP. Every added segment directly expands the physical collision footprint of the trailing chain.

### 8. Scoring

- **8.1 [REFINED]** The total final session score is calculated as a cumulative sum of all completed individual level scores at the terminal Game Over state.
- **8.2 [REFINED]** Level runtime tracking utilizes a continuous decimal millisecond timer that ticks while the level is active.
- **8.3 [REFINED]** The calculation formulas are explicitly defined as:

  ```
  level_time_bonus = max(0, (Max_Level_Time − level_time_seconds))
  level_score      = (level_target_value_sum × value_weight) + (level_time_bonus × time_weight)
  final_score      = Σ level_score_i  [for all completed levels]
  ```

- **8.4 [REFINED]** Default scoring system configuration values:
  - `value_weight` = 0.6
  - `time_weight`  = 0.4
  - `Max_Level_Time` = 300 seconds

### 9. Power-ups & Monetisation

- **9.1 [REFINED]** All power-ups, extra lives, active shields, path-finding assists, purchase storefronts, in-app transactions, or monetisation networks are entirely out of scope for this MVP iteration.

### 10. Legal / Ethical Compliance

- **10.1 [REFINED]** The use of real public individual names and operational agency statuses (Interpol / FBI) is restricted exclusively to local, non-public staging builds for testing purposes.
- **10.2 [REFINED]** A formal legal evaluation checking for defamation risk, right-of-publicity exposure, and app storefront distribution terms must serve as a mandatory gate blocking public or commercial deployment. Production environments must transition to entirely fictionalized asset packages.
